# S6 — Финальная интеграция и полировка

## Краткое описание
Сессия S6 — завершающая. Цель: полная сквозная интеграция сервера и клиента, устранение edge cases, улучшение UI/UX через ttk-стили и темы, финальная конфигурация деплоя, написание README и документации проекта.

---

## Стек технологий
| Компонент        | Технология                                      |
|------------------|-------------------------------------------------|
| Бэкенд           | FastAPI + Uvicorn (production config)           |
| База данных      | SQLite (финальный dev) / PostgreSQL (prod)      |

| Деплой           | systemd-сервис / Docker (опционально)           |
| Документация     | README.md + docstrings + Swagger UI             |

## Пошаговый план реализации

### ═══ ИНТЕГРАЦИЯ И ТЕСТИРОВАНИЕ ═══

### Шаг 1 — Полный сквозной сценарий (E2E checklist)
Пройти все сценарии вручную:
- [ ] Регистрация нового пользователя → логин → просмотр товаров → создание заказа → просмотр заказов → выход
- [ ] Логин как admin → открыть admin-панель → изменить роль пользователя → просмотреть статистику → выйти
- [ ] Попытка доступа к admin-функциям обычным пользователем → получить 403
- [ ] Создать заказ на товар с `stock=1`, затем снова → убедиться в корректной обработке
- [ ] Отключить сервер, попробовать действия в клиенте → все ошибки корректно обработаны

### Шаг 2 — Написание автотестов сервера
- [ ] Установить: `pip install pytest httpx`
- [ ] `conftest.py`:
  - Фикстура `test_client` — `TestClient(app)` с тестовой SQLite БД в памяти
  - Фикстура `auth_headers` — токен обычного пользователя
  - Фикстура `admin_headers` — токен admin-пользователя
- [ ] `test_auth.py`:
  - Регистрация с корректными данными → 201
  - Регистрация с дублирующимся username → 400
  - Логин с верными данными → 200 + токен
  - Логин с неверным паролем → 401
- [ ] `test_products.py`:
  - GET список (публичный) → 200
  - POST создать товар как admin → 201
  - POST создать товар как user → 403
  - PUT обновить товар → 200
  - DELETE удалить → 204
- [ ] `test_orders.py`:
  - POST создать заказ → 201, проверить автоподсчёт суммы
  - GET список заказов — user видит только свои
  - Admin видит все заказы

### Шаг 3 — Исправление edge cases
- [ ] Попытка создать заказ на несуществующий товар → `404 Not Found`
- [ ] Попытка заказать количество больше чем `stock` → `400 Bad Request` с понятным сообщением
- [ ] При создании заказа уменьшать `product.stock` на `quantity`
- [ ] При удалении/отмене заказа возвращать `stock` (если нужно по логике)
- [ ] Удаление пользователя — каскадное удаление его заказов или запрет удаления
- [ ] Пустые строки в полях формы — не отправлять на сервер
- [ ] XSS-безопасность: Pydantic автоматически валидирует типы, дополнительно ограничить `max_length`
- [ ] Истечение JWT токена — при получении 401 показать логин, не крашить приложение

## Ключевые технические решения

- **pydantic-settings**: удобное чтение `.env` с валидацией типов — лучше чем `os.getenv()`
- **ttk.Style + clam тема**: даёт возможность полного кастомизирования внешнего вида без сторонних библиотек
- **pytest + httpx**: `httpx.AsyncClient` или `starlette.testclient.TestClient` — стандарт для тестирования FastAPI
- **Каскадное удаление**: настроить `cascade="all, delete-orphan"` в SQLAlchemy relationships
- **Unicode иконки**: простой способ добавить визуальные подсказки без PIL/Pillow
- **Тестовая БД в памяти**: `sqlite:///:memory:` в conftest — тесты не трогают реальные данные
